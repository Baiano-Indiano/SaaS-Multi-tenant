# Plan 44-01: Smart Anomaly Detection & Alertas

**Goal:** Implement a lightweight heuristic anomaly scanner and proactive security alerts. This includes real-time MFA failure trackers on Redis, API catch-all interceptors, a background webhook surge scanner cron route, cooldown mechanics, admin notification dispatching via Resend, and robust Unit testing.

<threat_model>
- **Threat 1: Session Hijacking or Brute-Force Alert Bypass**
  - *Mitigation:* Ensure that verification route wrappers do not block user flows. Anomaly tracking must execute asynchronously/non-blocking to prevent DDoS vectoring on API responses.
- **Threat 2: Unauthenticated Cron Trigger (DoS on Anomaly Engine)**
  - *Mitigation:* Restrict the `/api/cron/anomaly-detector` endpoint with `CRON_SECRET` authorization header check.
- **Threat 3: Resource Exhaustion / Spamming Admin Inboxes**
  - *Mitigation:* Enforce a strict 30-minute Redis cooldown per alert type per tenant (`org:${orgId}:anomaly_cooldown:${type}`) before sending emails.
</threat_model>

## Tasks

### Wave 1: MFA Failure Tracking & Interception

<task id="44-01-01" name="Implement MFA failure tracking helper">
  <action>
    Create file `src/lib/security/mfa-tracker.ts`.
    Implement `trackMfaFailure(userId: string): Promise<void>`:
    1. Query organizations the user belongs to from `public.member` database table.
    2. For each organization, increment two Redis keys:
       - Janela curta: `org:${orgId}:mfa_failures_5m:${current_minute}` with 2 hours expiration.
       - Janela longa (limiar secundário): `org:${orgId}:mfa_failures_24h` (using INCR + EXPIRE if key is new) with 24 hours expiration.
    3. Sum counts of `org:${orgId}:mfa_failures_5m:*` for the last 5 minutes.
    4. Trigger anomaly alert `triggerAnomalyAlert(orgId, "MFA_SPIKE", ...)` if:
       - Short window sum > 10.
       - Long window count > 30.
  </action>
  <acceptance_criteria>
    - Helper exists and handles both Redis windows correctly.
    - Exceeding short or long thresholds triggers the alert pipeline.
  </acceptance_criteria>
</task>

<task id="44-01-04" name="Implement anomaly alert dispatcher with cooldown checks">
  <action>
    Create file `src/lib/security/anomaly-trigger.ts`.
    Implement `triggerAnomalyAlert(organizationId: string, type: "MFA_SPIKE" | "WEBHOOK_SURGE", details: string): Promise<void>`:
    1. Check cooldown in Redis: `org:${organizationId}:anomaly_cooldown:${type}`. If exists, log and abort.
    2. Set cooldown in Redis with 30 minutes expiration (1800 seconds).
    3. Fetch all admins and owners of the organization from database (role = 'admin' or role = 'owner').
    4. If the query returns no active admins/owners (e.g., orphan tenant), fallback to email list: `['security@saas-starter.internal']`.
    5. For each recipient, call `sendAnomalyAlertEmail`.
    6. Write a system audit log entry: `recordAuditLog({ organizationId, action: "SECURITY_ANOMALY_DETECTED", entityType: "ORGANIZATION", ... })`.
  </action>
  <acceptance_criteria>
    - Alert dispatcher implements the cooldown successfully.
    - Logs audit events into the tenant's schema under administrative bypass.
    - Fallback mechanism correctly alerts `security@saas-starter.internal` when no admins are present.
  </acceptance_criteria>
</task>

  </action>
