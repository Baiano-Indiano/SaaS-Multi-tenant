# Plan 42-01: Dynamic Tier-Based Rate Limiting

**Goal:** Implement dynamic tier-based rate limiting in the L1 proxy using Upstash Redis, updating organizations' cache with write-through logic on Stripe webhooks and server actions.

<threat_model>
- **Threat 1: Rate Limiter Bypass via Cache Poisoning or Outdated Cache**
  - *Mitigation:* Ensure that Stripe Webhooks perform an immediate write-through update of both `org:${orgId}` and `org:${orgSlug}` in Redis and the local `l1Cache`. On cache miss, safely fall back to the cryptographically stored key metadata `keyData.plan`.
- **Threat 2: Denial of Service due to Redis Downtime**
  - *Mitigation:* If Redis fails during the org data lookup in `src/proxy.ts`, catch the error, log to Sentry, and return a clean HTTP 503 Service Unavailable, ensuring that rate limiting checks cannot be bypassed silently when Redis is down.
</threat_model>
